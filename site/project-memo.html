Project Memo · Executive Summary

Return-shipping insurance pricing engine

A 3-5 minute overview of the project objective, business problem, data limits, modelling approach, final recommendation, and production next steps.

Project objective

The project builds an actuarial pricing workflow for return-shipping insurance sold to e-commerce sellers. It is not a generic return prediction exercise. The goal is to estimate expected insurance loss, convert that expected loss into a commercial premium, and show how the rate basis would be monitored after launch.

The final deliverable is a portfolio-ready pricing framework: exposure definition, synthetic claims, baseline pricing, GLM pricing, seller credibility, loss ratio monitoring, stress testing, and challenger model governance.

Business problem

A platform offering return-shipping coverage needs premiums that respond to seller, product, route, and freight risk. Charging one flat rate is simple, but it can underprice riskier sellers and overprice safer ones.

The pricing question is therefore not simply, "Will this item be returned?" The better insurance question is, "What covered return-shipping loss should this exposure carry, and is the total premium enough to support the target loss ratio?"

The broader product design can be modular: return shipping, failed delivery or logistics exception, and reshipment shipping. The current implemented pricing workflow deliberately keeps the first version narrower and prices covered return shipping only, so the recommendation is clear and auditable.

Data limitation

The project uses the public Olist e-commerce dataset. That data contains order, product, seller, route, delivery, and freight information, but it does not contain real return-shipping insurance claims.

A synthetic returns and claims layer is created so the pricing workflow can be tested end to end. This makes the results useful as a demonstration of actuarial judgement and model design, but not as production TikTok Shop rate indications. Production pricing would need real policy, return, claim, payment, expense, and time-period data.

The synthetic claim definition also excludes product refund, fraud, refund without return, failed delivery, and reshipment cost from the current rate indication. Those modules should be added only when real operational data supports them.

Method overview

The workflow starts by defining one exposure as a seller's product line within an order. That level is important because freight value and seller attribution live below the order level.

  1. Define the insurable exposure and coverage boundary.
  2. Create synthetic return and covered-claim outcomes.
  3. Estimate expected loss using frequency and severity.
  4. Apply seller credibility so noisy seller experience is not over-weighted.
  5. Backtest loss ratio, stress adverse scenarios, and compare a challenger model.

Core modelling approach

GLM is used as the champion pricing model because it gives a stable and explainable expected-loss basis. Seller credibility is then layered on top, so credible seller experience can affect price while small sellers remain close to the portfolio base.

XGBoost is retained as a challenger model. It is useful for testing whether a more flexible model finds additional risk ranking signal, but it does not replace GLM in the current recommendation because rate-level calibration matters more than a tiny improvement in ranking.

Key results

The workflow produces a coherent first-version pricing basis. The strongest evidence is that GLM calibration is close to actual loss, seller credibility preserves the total expected loss while adjusting seller-level risk, and stress testing identifies the main downside scenario.

Final recommendation

Use GLM expected loss plus seller credibility as the current pricing basis. This is the most mature recommendation because it balances calibration, explainability, seller differentiation, and operational monitoring.

Do not use XGBoost as the production pricing basis in the current version. Its test frequency AUC is slightly higher than GLM, but its pure premium A/E and loss ratio calibration are weaker. In insurance pricing, a better ranking score is not enough to replace a better-calibrated rate basis.

Main limitation and production next steps

The main limitation is that claims are synthetic. The project demonstrates the correct pricing architecture and governance logic, but production use would require real claim experience and out-of-time validation.

  1. Replace synthetic claims with real order, return, claim, and payment records.
  2. Build training, validation, and monitoring windows based on historical time.
  3. Add expense, margin, capital, reinsurance, and compliance assumptions.
  4. Set dynamic pricing guardrails: monthly rate refresh, sample-size thresholds, and single-change caps.
  5. Escalate sellers with credible loss ratios above risk thresholds to watchlist, manual review, or limit rules.
  6. Keep XGBoost in challenger governance until it proves stronger calibration.
